# roasting

> `/roasting`은 한국어 화이트칼라 산출물을 다섯 사람이 함께 다듬는 Claude Code 플러그인입니다.
> 월요일 아침에 보낼 거래처 이메일 한 통이 5분 안에 9.5점에 닿습니다. 임원 PPT 10장, 이사회 메모 1페이지, 랜딩페이지 한 화면, 인스타 릴스 커버 이미지 프롬프트 한 세트, 단일 HTML 웹사이트 한 페이지도 같은 합격선으로 다듬어 드립니다.
> 이메일부터 이사회 보고서·랜딩페이지·이미지 프롬프트·DART 회사 분석까지 68개 케이스를 자동으로 골라, 단발 도구가 아니라 **반복 가능한 운영체계**로 작동합니다.

---

## 1. 시작 (5분)

**설치는 두 줄입니다.**

```bash
/plugin marketplace add airoasting/roasting
/plugin install roasting@airoasting
```

**첫 호출은 자연어 한 줄이면 됩니다.**

```
/roasting 거래처 부장한테 8월 12일까지 회신 달라는 메일
```

이 한 줄을 넣으면 `/roasting`이 알맞은 케이스를 찾습니다. 케이스에 맞춰 다섯 사람이 캐스팅됩니다. 한 사람이 초안을 짓고, 네 사람이 각각 다른 관점에서 평가합니다. 합격선 9.5점에 닿을 때까지 최대 4라운드를 돕니다.

결과는 Claude Code 대화창과 세션 폴더(`~/.claude/roasting/_workspace/...`)에 함께 남습니다.

**산출물은 이렇게 나옵니다.**

위 한 줄을 입력하면 BLACK이 다음과 같은 초안을 짓고, 네 사람의 평가를 거쳐 합격선 9.5점에 닿을 때까지 다듬어집니다.

```
김OO 부장님,

지난 협의 건 답변을 8월 12일(월) 18시까지 회신 부탁드립니다.
회신이 늦어지면 다음 본부장 회의 안건 상정이 어려워, 동일 사안을
9월로 이연해야 합니다.

답변이 어려운 경우 일정 조정도 가능합니다. 한 줄로 알려주십시오.

[발신자 / 직책 / 회사]
```

이 초안에 RED 9.5 / SILVER 9.6 / BLUE 9.4 / GOLD 9.6이 매겨졌습니다. 네 사람의 평가 코멘트는 `critique.md`에, BLACK의 결정 로그는 `reasoning.md`에 함께 저장됩니다.

만약 4라운드를 돌아도 9.5점에 닿지 못하면 다음과 같이 사실대로 보고합니다.

```
완료. 9.5 미달 (max=9.2, round 4 forced output).
원인 가능성: BLUE 8.9. 거래처 부장의 호흡에 비해 마감일 압박이 직설적.
다음 액션: 캐스팅 또는 과제 정의를 다시 보세요. 1on1 권장.
```

기준 미달은 숨기지 않습니다. 그 자체가 사용자에게 정보입니다.

## 2. 이게 무엇인가

세 가지가 맞물려 돌아갑니다.

| 구성 요소 | 역할 |
|---|---|
| **5color 사이트** | 64개 원본 케이스 정의 (airoasting.github.io/5color) — 로컬에는 4개 확장 케이스(p41 임원 PPT · p70 랜딩페이지 · p75 DART 회사 분석 · p76 전략 메모)가 더 붙어 총 68개 |
| **slide_library** | PPT·랜딩 케이스용 슬라이드 템플릿 35종 (`slides`·`landing` 모드만 사용. p73 웹사이트 제작 `direct` 모드는 템플릿 없이 BLACK이 단일 HTML 직접 작성) |
| **/roasting** | 자동 분류 + 다섯 사람 실행 엔진 |

사용자는 한 줄만 던집니다. 분류·캐스팅·작성·평가·재작성은 `/roasting`이 처리합니다.

## 3. 이 방법론의 특성

5-Color Harness는 *다섯 사람이 한 산출물을 함께 만든다*는 한 줄로 요약됩니다. 단순한 'AI 글쓰기 보조'와 다른 점은 여섯 가지입니다.

### 3.1 **케이스별 캐스팅이 다릅니다**

이메일을 쓸 때의 다섯 사람과 임원 PPT를 만들 때의 다섯 사람은 같지 않습니다. 케이스마다 BLACK 수행자는 다른 페르소나로 캐스팅됩니다.

| 케이스 | BLACK 캐스팅 예시 |
|---|---|
| 외부 비즈니스 이메일 | B2B SaaS 외부 협력 시니어 15년차+ (베인 컨설턴트 응답 메일 톤) |
| 임원용 PPT | 컨설팅펌 출신 사내 전략기획 디렉터 17년차+ (베인·맥킨지 임원 덱 톤) |
| 사과문 | 위기 커뮤니케이션 디렉터 18년차+ (PR 위기 대응 책상 톤) |
| 랜딩페이지 | 그로스 마케팅·랜딩 UX 카피라이터 17년차+ (실리콘밸리 SaaS 랜딩 톤) |

네 사람의 비평가도 마찬가지로 케이스마다 다른 어휘·다른 마무리 패턴·다른 합격선 장면을 갖습니다. 같은 RED라도 이메일에서 보는 것과 IR 발표에서 보는 것이 다릅니다.

### 3.2 **GOLD는 평가자가 아니라 *읽는 사람*입니다**

다른 비평가(RED·SILVER·BLUE)는 산출물을 평가하지만, GOLD는 *실제 수신자가 그 문서를 받는 장면*을 시뮬레이션합니다.

> 화요일 오전 9시, CEO가 임원회의 직전에 덱을 5분에 훑는다.

이 장면이 케이스마다 구체적으로 정의되어 있습니다. GOLD는 그 장면 안에서 "이 문서를 받았을 때 사인을 할 것인가, 다음 질문을 할 것인가, 닫아버릴 것인가"를 봅니다. 점수가 아니라 *체류도*를 봅니다.

대부분의 AI 비평 도구는 "이 글이 좋은가"를 묻습니다. GOLD는 "이 글로 수신자가 다음 행동을 할 것인가"를 묻습니다.

### 3.3 **비평가들은 일부러 의견이 갈립니다**

네 사람의 점수 편차(σ)가 0.5 이상 벌어지면 자동으로 토론이 시작됩니다. 가장 높게 준 사람과 가장 낮게 준 사람이 짧게 코멘트를 주고받고 다시 채점합니다. 억지로 합의를 만들지는 않습니다. 토론을 거친 뒤에도 의견이 갈리면 그 상태 그대로 사용자에게 보여줍니다.

다른 도구는 보통 평균을 내고 끝냅니다. 5-Color는 *합의가 쉬울 때만 평균*을 씁니다. 합의가 어려우면 그 자리에서 갈등을 드러냅니다.

### 3.4 **합격선이 숫자로 박혀 있습니다**

"적당히 괜찮은 정도"가 아닙니다. 분석 3인 평균 9.5 이상, GOLD 체류도 9.5 이상이 합격선입니다. 4라운드를 돌아도 못 넘으면 강제 종료하고 사용자에게 보고합니다.

> "9.5 미달 (max=9.2). 캐스팅 또는 과제 정의 문제 가능성. 1on1 권장."

기준 미달을 *숨기지 않습니다*. 기준 미달이라는 사실이 그 자체로 사용자에게 정보가 됩니다.

### 3.5 **5가지 안티패턴은 평가 전에 자동으로 잡힙니다**

비평가 4인이 채점하기 전에 다섯 가지 안티패턴 검출기가 먼저 돕니다.

| 안티패턴 | 예시 |
|---|---|
| `VAGUE_CTA` | "검토 부탁드립니다"로 끝나는 메일 (무엇을 결정할지 모름) |
| `LEGAL_RISK_TERM` | "절대로", "100% 보장합니다" 같은 단정 약속어 |
| `HALLUCINATED_NUMBER` | 출처 없는 "20% 성장", "1,200개사" 같은 숫자 |
| `MISSING_GOLD_HOOK` | 첫 200자에 GOLD 합격선 장면이 안 살아있는 경우 |
| `TONE_MISMATCH` | 케이스 캐스팅 톤과 첫 문장 어휘가 어긋난 경우 |

검출되면 BLACK이 *비평가에게 가기 전에* 한 번 스스로 고쳐 씁니다. 라운드 카운트는 깎이지 않습니다. 같은 안티패턴이 3회 연속 잡히면 사용자에게 보고합니다.

### 3.6 **68개 케이스가 곧 IP입니다**

이 도구가 만드는 가치는 "AI가 글을 잘 쓴다"가 아닙니다. *어떤 케이스에 누구를 캐스팅해야 하는지*가 가치입니다. 68개 케이스마다 BLACK·RED·SILVER·BLUE·GOLD 다섯 사람의 페르소나·합격선 장면·금지 표현·문체 지침이 명시되어 있습니다.

이 정의 자체가 `references/cases/p1.md` ~ `p76.md` 파일로 박혀 있고, 5color 사이트(airoasting.github.io/5color)에 64개 원본이 공개되어 있습니다. 나머지 4개(p41 임원 PPT, p70 랜딩페이지, p75 DART, p76 전략 메모)는 로컬 확장 케이스입니다.

## 4. 다른 하네스 스킬과 무엇이 다른가

"하네스(harness)"는 LLM 위에 *역할·규칙·검증 절차*를 얹어 단발 출력을 운영체계로 만드는 접근입니다. 자주 보는 패턴과 5-Color의 차이는 아래 표와 같습니다.

| 비교 대상 | 일반적인 방식 | 5-Color Harness |
|---|---|---|
| **Director-Critic (감독-비평)** | 감독 한 명이 생성, 비평 한 명이 평가 | 네 명이 *서로 다른 축*으로 동시 평가, σ ≥ 0.5면 토론 |
| **Self-Refine / CoT** | 한 모델이 자기 출력을 보고 다시 쓰는 1인 루프 | 다섯 페르소나가 *역할 분리* 상태로 통신, 페르소나 침범 금지 |
| **Multi-Agent Debate** | 동질한 에이전트 2~3명이 같은 문제를 토론 | 이질적 4축(이성·전문성·공감·독자 시나리오)이 다른 어휘로 평가 |
| **Constitutional AI** | 규칙 기반 자기 비판 (룰 헌법) | 케이스별 *페르소나*와 *장면*이 비판 기준 |
| **Producer-Reviewer (일반)** | 1+1, 평균 또는 통과/실패 | 1+4, 9.5 합격선 + 4라운드 cap + 안티패턴 사전 검출 |
| **RLHF / RLAIF** | 학습 단계의 피드백 신호 | 추론 시점의 인-라인 비평, 사용자 호출당 작동 |

핵심 차이는 *비평가가 누구냐*입니다. 다른 하네스는 보통 *범용 평가자*를 둡니다. 5-Color는 *수신자가 처한 장면*과 *케이스가 요구하는 직업적 안목*을 비평가의 정체성으로 만듭니다. "이 글이 좋은가"가 아니라 "*이 사람이 이 장면에서 이 글을 받았을 때*"를 묻습니다.

또 한 가지 차이는 *비평을 시작 전에 한 번 잘라낸다*는 점입니다. 5가지 안티패턴 검출기가 BLACK 산출물을 먼저 훑고, 명백한 결함(모호한 CTA·출처 없는 수치 등)은 비평가에게 가기 전에 BLACK이 스스로 고칩니다. 비평가는 *수치 누락 같은 1차 결함*이 아니라 *논리·구조·톤·체류도* 같은 2차 결함에만 집중합니다.

## 6. 다섯 사람의 역할

다섯 사람이 산출물 하나를 함께 만들고 평가합니다.

| 역할 | 담당 관점 | 한 줄 |
|---|---|---|
| **BLACK** | 수행자 | 케이스에 맞는 전문가 페르소나로 산출물을 짓습니다 |
| **RED** | 이성·논리 | 의도가 한 줄로 전달되는지, 논리가 단단한지 봅니다 |
| **SILVER** | 분야 전문가 | 해당 문서 유형의 구조와 관행에 맞는지 짚습니다 |
| **BLUE** | 공감·톤 | 읽는 사람 입장에서 톤과 분량이 적절한지 봅니다 |
| **GOLD** | 독자 시나리오 | 실제 수신자가 이 문서를 받는 순간을 시뮬레이션합니다 |

**합격 기준:**
- 분석 3인(RED·SILVER·BLUE) 평균 9.5 이상
- GOLD 체류도 9.5 이상
- 최대 4라운드 반복
- 4라운드를 돌아도 미달이면 강제 출력 + 사유 보고

네 사람의 점수가 0.5점 이상 벌어지면 그 자리에서 짧게 토론을 합니다. 가장 높은 점수와 가장 낮은 점수가 코멘트를 주고받고 새 점수를 매깁니다. 한 라운드의 토론이 끝나도 합의가 안 되면 합의 실패로 표시합니다.

**합격선이 왜 9.5인가.** 모든 케이스에 같은 숫자를 쓰지 않습니다. 작업 영역마다 결함의 무게가 다르기 때문입니다.

| 합격선 | 적용 영역 | 이유 |
|---|---|---|
| **9.0** | 시·단편 소설·블로그 같은 *개인 톤 자산* 영역 | 정답이 없고 작가의 호흡이 자산입니다. 정량 강제가 톤을 깎습니다. |
| **9.5** (기본) | 이메일·보고서·사내 공지 같은 *표준 화이트칼라* 영역 | 베타 사용자 데이터에서 9.5 이상이면 *수신자가 의도대로 행동*하는 비율이 90%를 넘습니다. |
| **9.7** | 계약서·이사회 보고서·주주 서한·사과문·IR 발표 같은 *결함이 곧 손실*인 영역 | 한 줄의 단정 약속어가 법적 노출을 만들고, 한 단어의 변동이 신뢰를 잃습니다. |

케이스 정의에 합격선이 명시되어 있고, BLACK 캐스팅 줄에 같이 박혀 있습니다.

## 7. HTML 산출물 케이스: 3-mode 빌더

HTML로 떨어지는 케이스는 빌더 모드가 셋입니다. `mode_for_case()`가 단일 진실로 분기합니다.

| 모드 | 케이스 | 작동 방식 |
|---|---|---|
| **`slides`** | p41 임원 PPT, p42 강의 자료, p43 영업 덱, p45 컨설팅 덱 | `slide_library` 35개 템플릿 중 가장 적정한 1개를 골라 베이스 HTML로 사용. BLACK 산출물을 슬라이드 단위(H1·H2)로 주입. |
| **`landing`** | p70 웹사이트·랜딩페이지 | 같은 35개 템플릿 중 1개 선택. 차이는 분할 방식 — H2 단위로 랜딩 섹션(히어로·소셜프루프·기능·CTA·FAQ 등)을 주입. |
| **`direct`** | p73 웹사이트 제작 | **템플릿 사용 안 함.** BLACK이 처음부터 단일 HTML 파일(인라인 CSS · 시맨틱 마크업 · 반응형 · WCAG 2.1 AA)을 직접 작성. 빌더는 BLACK 출력을 `output.html`로 그대로 통과시킴. |

```
/roasting 이번 분기 임원 PPT (실적 + 리스크 + 결정사항 3개)        # slides 모드
/roasting B2B SaaS 결제 모듈 랜딩페이지 만들어줘                  # landing 모드
/roasting Stripe 스타일 SaaS 한 페이지 사이트 만들어줘            # direct 모드
```

`slides`·`landing` 모드 흐름은 다음과 같습니다.

1. `/roasting`이 케이스를 분류합니다 (예: `p41` 임원 PPT, `p70` 랜딩페이지).
2. `slide_library`가 색상·격식·톤 기준으로 후보 템플릿 3개를 추천합니다.
3. 사용자가 그중 하나를 고르면 그 선택을 케이스별 선호로 저장합니다.
4. BLACK이 그 템플릿 HTML을 베이스로 산출물을 짓습니다.
5. PPT는 슬라이드 단위, 랜딩은 H2 섹션 단위로 콘텐츠가 주입됩니다.
6. 네 사람의 평가가 끝나면 최종 HTML 파일이 남습니다.

`direct` 모드(p73)는 위 흐름에서 2·3·5단계를 건너뜁니다. 템플릿 추천도, 컨테이너 주입도 없습니다. BLACK이 곧 디자이너이자 마크업 작성자입니다.

같은 케이스를 다시 호출하면 저장된 템플릿이 자동으로 적용됩니다. 다른 템플릿을 쓰고 싶으면 사용자가 명시적으로 바꾸면 됩니다. `direct` 모드는 템플릿이 없으므로 이 단계가 무의미합니다.

## 8. 다른 스킬과의 결합 (`enrich` 필드)

11개 케이스가 다른 Claude Code 스킬을 자동으로 호출합니다. BLACK이 산출물을 짓기 전에 컨텍스트를 보강하는 단계입니다.

| 케이스 | 호출 스킬 | 조건 |
|---|---|---|
| p25 기획안, p45 컨설팅 덱, p37 산업 분석, p40 경영 진단, p76 전략 메모 | `/strategy` | 전략적 질문이 감지된 경우 |
| p28 이사회 보고서, p29 주주 서한, p33 분기 실적, p75 DART 회사 분석 | `/dart` | 회사명·티커가 감지된 경우 |

해당 스킬이 설치되어 있지 않으면 조용히 건너뛰고 일반 모드로 진행합니다. 사용자가 따로 챙길 일이 없습니다.

## 12. 알려진 한계

**v0.3 베타 기준입니다.**

- **베타 라벨:** 케이스 품질과 평가 기준은 계속 다듬어 갑니다.
- **Agent Teams 실험성:** 평가자 4명이 동시에 도는 경로는 Anthropic 실험 기능을 씁니다. 오류가 나면 순차 모드로 자동 전환됩니다.
- **자동 sync 미지원:** 5color 사이트가 업데이트되면 `make sync-cases`를 한 번 돌리면 됩니다. 로컬 확장 케이스(p41 · p70 · p75 · p76)는 sync에 영향받지 않고 보호됩니다.
- **한국어 전용:** 케이스와 평가 기준 모두 한국어로 작업합니다. 영어 입력이 들어오면 자산 시드 없이 일반 5-Color 모드로 폴백합니다.
- **HTML 슬라이드:** PPT 4종은 HTML 형식으로 출력합니다. PowerPoint(.pptx) 직접 출력은 v0.4에서 추가됩니다. p73 웹사이트 제작은 단일 HTML 직접 출력으로 v0.3에서 추가됐습니다.
- **Enrich 호출:** `/dart`·`/strategy` 실제 dispatch는 v0.4에 들어갑니다. v0.3은 메타데이터만 준비합니다.
